
# 步态分析算法模块数据接口与对接协议

**2026.3.24**

## 一、背景

为便于步态分析算法模块顺利交付和集成，对算法模块的输入数据、输出数据、调用方式、文件格式及异常处理方式进行统一约定。

步态分析算法端采用 Python 编写并封装为本地可执行程序，上位机采用 C# WPF 编写和集成。本协议用于规范上位机向算法模块发起分析请求、获取算法分析状态与结果、读取和显示步态参数及运动学参数的全过程。

---

## 二、数据

对受试者行走视频进行分析，提取步态事件参数和运动学参数，并输出相应结果，用于上位机展示、报告生成和评估分析。

### 1. 输入

输入数据包括双相机拍摄视频、受试者信息和算法运行参数。
其中：
- 视频存放于本地固定目录，由算法模块按输入路径直接读取；
- 受试者信息、任务信息、设备信息、分析选项及输出路径由上位机打包为 JSON 配置文件，随启动指令一并传递给算法模块；
- 算法运行所需关键参数至少包括受试者身高、侧向相机与患者步道间的距离。
### 2. 输出
输出数据包括步态事件参数、运动学参数、时间序列数据文件及任务执行信息。

（1）步态事件参数  
包括：
- 步态周期时长
- 步长
- 步幅
- 步频
- 步速
- 站立相时间
- 摆动相时间
- 双支撑相时间
- 单支撑相时间

（2）运动学参数  
包括：

- 关节角度活动范围（ROM）
- 髋关节在矢状面的角度—时间曲线
- 膝关节在矢状面的角度—时间曲线
- 踝关节在矢状面的角度—时间曲线
- 骨盆在冠状面的角度—时间曲线
- 关键点运动轨迹、速度、角速度曲线

其中，步态事件参数和运动学参数中的汇总结果统一输出为 JSON 文件；角度—时间序列、关键点轨迹、速度、角速度等逐帧数据统一输出为 CSV 文件。每次分析任务单独生成一个输出目录，避免多次分析结果混淆。

汇总结果 JSON 中统一包含：

- 任务状态
- 是否成功
- 错误码
- 错误信息
- 生成时间
- 输出目录及文件路径
- 步态事件参数汇总结果
- 运动学参数汇总结果
- 质量控制信息
---

## 三、调用方式

本协议采用**本地程序调用方式**。

算法模块提供本地可执行程序exe入口，由 WPF 上位机通过启动进程方式调用。上位机向算法模块传入任务配置文件路径，算法模块执行完成后输出 JSON 结果文件和 CSV 数据文件，上位机读取结果文件并展示。

调用命令格式示例如下：

```bash
gait_analysis.exe --config "D:\GaitTasks\task_20260324_001.json"
```

其中：

- `gait_analysis.exe` 为演示的算法模块可执行程序；
- `--config` 为任务配置文件路径；
- 配置文件采用 JSON 格式；
- 视频路径、输出目录路径均在配置文件中给出；
- 算法模块运行过程中通过标准输出返回任务状态；
- 算法模块运行结束后通过结果 JSON 和进程退出码共同表示任务执行结果。

## 四、输入通讯内容

### 4.1 输入内容

算法模块接收的输入信息包括以下内容：
#### （1）基本任务信息

- 请求编号--request_id
- 协议版本号--protocol_version
- 算法版本号--algorithm_version
- 模型版本号--model_version
- 分析任务类型--task_type
- 分析模式--analysis_mode

#### （2）受试者信息

- 受试者编号--subject_id
- 性别--gender
- 年龄--age
- 身高（重要）--height_m
- 体重--weight_kg

其中，受试者编号、身高为必填项。
#### （3）视频信息

- 侧向视频文件路径--sagittal_video_path
- 正向视频文件路径--coronal_video_path
- 视频帧率--video_fps
- 视频分辨率--video_resolution
- 起始分析时间点--start_time_s
- 分析时长--duration_s

其中，双视频文件路径为必填项；起始分析时间点和分析时长用于限定分析区间。

#### （4）设备信息

- 相机编号--camera_id
- 相机类型--camera_type
- 采集帧率--capture_fps
- 侧向相机与患者步道间的距离（重要）--side_camera_to_walkway_distance_m
- （其他必要采集参数）

其中，侧向相机与患者步道间的距离为必填项。

#### （5）分析选项

- 是否计算步态事件参数--calculate_gait_event_parameters
- 是否计算运动学参数--calculate_kinematic_parameters
- 是否导出 CSV 文件--export_csv
- 是否进行曲线平滑处理--smooth_curve

输出单位统一如下：

- 时间单位：秒（s）
- 长度单位：米（m）
- 角度单位：度（°）
- 速度单位：m/s
- 角速度单位：°/s

## 五、输出通讯内容
### 5.1 输出内容

算法模块完成分析后，输出以下内容：

#### （1）任务执行结果

- 任务编号--request_id
- 协议版本号--protocol_version
- 算法版本号--algorithm_version
- 模型版本号--model_version
- 任务状态--task_status
- 是否成功--success
- 错误码--error_code
- 错误信息--error_message
- 生成时间--generated_time
- 输出目录路径--output_dir

#### （2）步态事件参数结果

输出一组汇总参数，包括：

- 步态周期时长--gait_cycle_duration_s
- 步长--step_length_m
- 步幅--stride_length_m
- 步频--cadence_step_per_min
- 步速--gait_speed_m_per_s
- 站立相时间--stance_time_s
- 摆动相时间--swing_time_s
- 双支撑相时间--double_support_time_s
- 单支撑相时间--single_support_time_s

#### （3）运动学参数结果

输出一组汇总指标，包括：

- 髋关节 ROM--hip_rom_deg
- 膝关节 ROM--knee_rom_deg
- 踝关节 ROM--ankle_rom_deg
- 骨盆角度相关统计值--pelvis_coronal_rom_deg
- （其他必要运动学统计值）

#### （4）时间序列数据文件信息

输出本次分析生成的 CSV 文件路径，包括但不限于：

- 关节角度时间序列 CSV--joint_angle_csv
- 关键点轨迹 CSV--keypoint_trajectory_csv
- 关键点速度 CSV--keypoint_velocity_csv
- 关节角速度 CSV--joint_angular_velocity_csv

#### （5）质量控制信息

输出基础质量控制信息，包括但不限于：

- 关键点识别置信度--mean_keypoint_confidence
- 视频有效帧比例--valid_frame_ratio
- 是否存在遮挡--occlusion_warning
- 是否存在丢点--missing_point_warning
- （其他必要告警信息）

## 六、输出CSV数据

CSV 文件统一采用带表头格式输出
### 6.1 CSV数据结构

关节角度曲线、关键点轨迹、速度、角速度等逐帧或逐时间点变化的数据，统一保存为 CSV 文件，供上位机读取绘图、本地留存和人工核查。

每次分析结果单独生成一个输出目录，目录中包含：

- 汇总结果 JSON 文件
- 各类时间序列 CSV 文件
- 日志文件

目录结构如下：

```text
GAIT_20260324_001/
├─ summary.json
├─ joint_angle_timeseries.csv
├─ keypoint_trajectory.csv
├─ keypoint_velocity.csv
├─ joint_angular_velocity.csv
└─ debug.log
```

## 七、交互形式

每次分析任务按以下顺序完成交互：

1. 上位机生成任务配置 JSON 文件；
2. 上位机通过进程方式启动算法 exe，并传入配置文件路径；
3. 算法模块读取配置文件和视频路径；
4. 算法模块在执行过程中通过 stdout 输出任务状态；
5. 算法模块完成分析后写出 `summary.json`、CSV 文件和日志文件；
6. 算法模块退出；
7. 上位机读取 `summary.json` 及各 CSV 文件并进行展示。

任务状态消息统一采用单行 JSON，通过 stdout 输出，格式如下：

```json
{
  "type": "status",
  "request_id": "GAIT_20260324_001",
  "task_status": "running",
  "progress": 35,
  "message": "detecting keypoints"
}
```

状态取值统一为：

- `pending`      待执行
- `running`      执行中
- `completed`   已完成
- `failed`        执行失败
- `cancelled`   已取消
-
状态消息示例如下。

任务开始：
```json
{"type":"status","request_id":"GAIT_20260324_001","task_status":"pending","progress":0,"message":"task accepted"}
```

执行中：
```json
{"type":"status","request_id":"GAIT_20260324_001","task_status":"running","progress":35,"message":"detecting keypoints"}
```

分析完成：
```json
{"type":"status","request_id":"GAIT_20260324_001","task_status":"completed","progress":100,"message":"analysis completed"}
```

分析失败：
```json
{"type":"status","request_id":"GAIT_20260324_001","task_status":"failed","progress":100,"message":"video file not found","error_code":2}
```

## 八、异常处理

任务结果判断采用两三层信息机制来获取
1. `summary.json` 中记录的算法分析状态；
2. 日志信息；
3. exe 进程退出码。
其中以 `summary.json` 中状态为主，日志信息和exe 退出码为辅，综合判断任务是否成功。
### 1.错误信息
如执行失败，算法模块返回：

- 错误码
- 错误说明
- 失败原因简述

错误场景包括但不限于：

- 视频文件不存在
- 视频格式不支持
- 视频读取失败
- 关键点识别失败
- 步态事件识别失败
- 结果导出失败

### 2.日志信息
日志文件统一输出到：
```text
debug.log
```

日志内容包括：
- 启动时间
- 输入参数信息
- 分析过程关键步骤
- 异常报错信息
- 输出文件生成情况
### 3.进程退出码
进程退出码统一如下：

|退出码|含义|
|---|---|
|0|执行成功|
|1|配置文件错误|
|2|输入文件不存在|
|3|视频读取失败|
|4|分析处理失败|
|5|结果导出失败|
|9|未知错误|

## 九、版本管理

为保证算法升级后与上位机保持兼容，输入信息中统一包含：

- 协议版本号
- 算法版本号
- 模型版本号

当字段发生新增或调整时，优先采用新增字段方式处理，保持既有字段兼容；如发生不兼容调整，则同步更新协议版本号。
