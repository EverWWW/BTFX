# 二、软件显示类规范
## 1. 图表规范
1.  **必备元素**：
    *   需清晰标注纵轴与横轴的标识。
    *   需明确标识柱形、线条等图形所代表的数据含义。
    *   必须标注数据单位。
2.  **显示与兼容性**：
    *   设计时需考虑打印场景，确保色块在黑白打印后仍可区分。
    *   当数据量过多时，应有明确的显示策略（例如：默认或设置项可选“仅显示最近N次记录”），以确保图表清晰、完整。

## 2. 报告规范
1.  **内容要素**：
    *   显示医院/单位的名称。
    *   **医生意见**部分需设计为可编辑字段。
    *   显示患者的基本信息（如姓名、性别、年龄等）。
    *   评估、治疗、训练等主要信息需完整、美观地陈列。
    *   评估报告的尾页需附上对应的评分标准。
2.  **格式与功能**：
    *   显示页码（格式如：第 X 页 / 共 Y 页）。
    *   所有数据需附带单位，并与其他页面保持一致。
    *   必须提供报告打印功能，并在设计时充分考虑打印后的版面显示效果（避免内容被截断、样式错乱）。

## 3. 记录列表规范
1.  **排列与排序**：
    *   列表字段的排列顺序需符合业务逻辑和用户习惯，经确认后固定。
    *   记录默认按照生成时间**倒序排列**（最新的在前）。
2.  **显示格式**：
    *   涉及时间、得分、年龄的字段，必须加上单位。
    *   字段名称和字段值均需显示完整，避免因长度不够而截断导致信息不全。

## 4. 排版与视觉规范
1.  **品牌元素**：
    *   使用公司最新要求的Logo，并遵循指定的配色、大小和位置规范。
    *   **禁止**使用带有 ®（已注册）或 ™（商标）标识的Logo版本。
2.  **界面元素**：
    *   按钮上的文案必须显示完整。
    *   状态栏需清晰显示当前系统状态或关键信息。
3.  **对齐与字体**：
    *   字段与标题需有统一、美观的对齐方式。
    *   整个界面使用的字体类型必须一致。
    *   标题的字号**不得小于**正文字号。

## 5. 系统整体风格规范
1.  遵循公司或项目定义的统一配色方案。
2.  各个功能模块的视觉风格（如颜色、间距、控件样式）需保持一致。
3.  全系统使用的字体类型、大小需统一。

## 6. 版本号显示规范
| 版本类型 | 版本号格式示例 | 显示位置与说明 |
| :--- | :--- | :--- |
| **发布版本** | V1 | 显示在主界面、登录界面、激活界面。 |
| **完整版本** | V1.0.0.1 | 在“关于我们”界面显示，或**点击主界面版本号可弹出查看**（具体方式根据项目定义）。 |
| **内测版本** | V1.0.0.20250729_alpha01 | 在“关于我们”界面显示，或**点击主界面版本号可弹出查看**，仅供测试使用。 |

## 7. 图表常见问题自查清单
开发与测试时，需重点检查以下问题点：
*   坐标轴缺失标识。
*   未严格按照UI设计稿进行排版。
*   图表元素（如数据标签、图例）发生重叠。
*   数据单位缺失。
*   前端显示的字段与后台关联的字段不一致。

## 8. 关键注意事项
1.  **中英文与字段统一性**：
    *   同一字段在不同界面、不同语境下的名称必须完全一致。
    *   **典型示例**：“电话号码”、“联系方式”、“电话”、“手机”、“手机号”等应统一为其中一个，全系统保持一致。
    *   **典型示例**：“姓名”、“名称”、“昵称”等应依据业务场景统一。
2.  **设备与连接状态管理**：
    *   **单用户设备**：同一设备若在物理上只能单人使用，则软件逻辑上禁止同时进行多项操作（如同时开始治疗和评估）。
    *   **多用户设备**：根据设备实际能力设计，但**同一用户**账号不能同时进行多项操作。
    *   软件正常关闭时，必须确保设备同步停止运行（强制断电、强制结束进程等非正常关闭除外）。
    *   当物理连接被断开（如拔掉串口/USB线）或客户端被关闭时，界面必须明确显示设备“已断开连接”的状态。
# 三、软件操作类规范

| 操作               | 验证方式                   | 备注与具体要求                                                                                                                                                                                 |
| :--------------- | :--------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **退出系统**         | 应有二次确认操作提示（弹窗）。        | 提示框的样式需注意，与图表等元素的视觉风格保持一致。                                                                                                                                                              |
| **退出登录**         | 直接操作，或根据项目要求决定是否需确认。   | -                                                                                                                                                                                       |
| **删除记录**         | 操作前应有确认提示。             | 提示信息需明确，例如“确认删除该条记录？删除后不可恢复。”                                                                                                                                                           |
| **删除系统自定义处方**    | 操作前应有确认提示。             | 1. 提示信息需明确。<br>2. **内置的默认处方不可删除**，应在逻辑上锁定或界面上禁用删除按钮。                                                                                                                                    |
| **重复打开软件**       | 检测程序是否已运行。             | 只允许打开一个软件实例。当尝试再次打开时，应提示“程序已运行，请勿重复启动。”（具体文案可根据项目定义）。                                                                                                                                   |
| **打开页面资源缺失/加载慢** | 常规监控与测试。               | 项目开发部署时需排查以下情况：<br>1. **不要使用CDN资源**：项目通常部署在内网，可能无法访问外网。<br>2. 确保CSS、JS文件引用正确，无缺失。<br>3. 确保图片等静态资源路径正确，能正常加载。                                                                            |
| **操作锁定**         | 对“确认”、“提交”等关键按钮进行点击控制。 | 快速多次点击时，只应生效一次，后续点击能被有效过滤或屏蔽，防止产生重复数据或请求。                                                                                                                                               |
| **系统提示**         | 对错误、警告等提示信息进行频率控制。     | 用户快速多次进行可能触发相同提示的操作时，系统在短时间内应只提示一次，避免提示信息“刷屏”，影响操作。                                                                                                                                     |
| **记录删除后页面跳转**    | 删除操作完成后检查列表状态。         | 当删除列表最后一页的全部数据后，页面应能自动跳转到上一页，不应停留在空页面。                                                                                                                                                  |
| **所有功能键**        | 全面测试。                  | 确保所有按钮、图标等可操作元素，在点击或触屏操作下均能正常响应。                                                                                                                                                        |
| **急停**           | 模拟软件运行时触发急停。           | 1. **评估/治疗/训练中急停**：上位机软件应立即弹出明确提示，且软件内部状态与下位机（设备）实际状态保持一致，均为“已急停”。<br>2. **释放急停**：释放后，上位机软件状态与下位机状态应同步恢复一致。（注：针对机器人等复杂项目，具体交互逻辑待定）。                                                     |
| **内置账号管理**       | 尝试删除或修改内置账号。           | 内置的系统默认账号（如 `admin`, `user`）**不可删除**。<br>关于密码：<br>1. **方案A**：允许修改密码，但需提供隐藏的管理员操作入口，可将其密码重置为默认密码。<br>2. **方案B**：禁止修改内置账号密码。<br>（具体采用哪种方案，需根据项目安全要求确定）。                                   |
| **安装**           | 执行安装程序。                | 应使用安装包（如MSI、Setup.exe），实现一键安装或有非常清晰明确的向导式安装提示。<br>**避免以下问题**：<br>1. 需要用户单独连接、配置数据库。<br>2. 需要用户手动配置端口等复杂参数。<br>3. 未提供自定义安装路径的选项。<br>4. 安装文件名称混乱、不统一。<br>5. 安装后生成的桌面快捷方式，其名称和图标应明确体现项目名称。 |
| **升级**           | 执行升级安装程序。              | 需根据升级内容，明确告知用户升级方式及影响：<br>1. **覆盖安装（数据可能清空）**：当数据库结构等有重大复杂变动时，需在升级前**明确提示用户**“本次升级将清空现有数据”，并让用户确认。<br>2. **增量安装（数据保留）**：当仅为功能更新，数据结构无复杂变动时，应支持直接覆盖安装，保留原有数据库和用户数据。                      |
| **激活**           | 测试在线和离线激活流程。           | 纯软件产品应同时具备在线激活和离线激活（如使用授权文件）两种功能，以适应不同的用户网络环境。                                                                                                                                          |
| **卸载**           | 通过系统控制面板或程序目录查找卸载方式。   | 必须提供完整的卸载功能（通过安装包或独立的Uninstall程序），确保能完整移除软件本体、相关服务及注册表项（如需），避免产生系统垃圾。                                                                                                                   |

| :--- | :--- |

| 缺陷描述 | 逻辑要求与备注 |
| :--- | :--- |
| 管理员和普通用户的权限没有区分。 | 必须实现清晰的权限分级：<br>1. **管理员**：拥有系统最高权限，可管理用户账号、系统设置等。<br>2. **医生用户**：拥有核心业务操作权限（如创建报告、管理患者）。<br>3. **普通用户**：根据项目定义，权限应受限。<br>**附加规则**：<br>- 医生角色的用户应能看到系统中**所有**的患者列表。<br>- 管理员可对医生账号进行创建、修改、禁用等管理操作。 |

## 6. 医生账号管理
| 缺陷描述 | 逻辑要求与备注 |
| :--- | :--- |
| 医生可以被删除，删除后，其创建的报告信息显示不全或出现异常。 | **禁止直接删除医生账号**。正确的逻辑是：<br>1. 管理员可以对医生账号进行**禁用**操作。<br>2. 禁用后，该医生无法登录系统。<br>3. 该医生已创建的报告、患者关联等历史数据必须完整保留，并正常显示。 |
| 删除医生后，其关联的患者数据被意外删除。 | **绝对禁止**删除医生时级联删除其关联的患者数据。患者数据是独立的核心资产。 |

## 8. 删除操作安全
| 缺陷描述 | 逻辑要求与备注 |
| :--- | :--- |
| 管理员账号可以被删除或修改角色后被删除，导致系统无管理员无法登录。 | 必须至少保留一个激活状态的管理员账号。系统应内置逻辑，防止最后一个有效管理员账号被删除或降权。 |
| 可以删除正在接受评估、治疗或训练的用户，导致相关功能界面报错或状态异常。 | 删除前必须进行状态检查：<br>1. 若用户正在业务操作中，应提示“该用户正在[评估/治疗/训练]中，无法删除，请先结束相关操作”。<br>2. 确认删除后，需确保所有相关界面和后台状态得到妥善清理，不出现空指针或数据不一致的错误。 |
