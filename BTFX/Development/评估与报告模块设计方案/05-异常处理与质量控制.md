# 05 - 异常处理与质量控制

## 一、异常处理分层策略

```
异常处理三层架构
│
├── Layer 1：算法层（Python exe 内部）
│   ├── 输出 summary.json 中的 error_code + error_message
│   ├── stdout 输出 failed 状态消息
│   ├── 进程退出码（0-9）
│   └── debug.log 日志文件
│
├── Layer 2：调用层（GaitAnalysisService）
│   ├── 进程启动失败（exe 不存在、权限不足）
│   ├── 进程超时（超过配置时间未完成）
│   ├── 进程异常退出（崩溃、被杀死）
│   ├── stdout 解析异常（非 JSON 格式）
│   ├── 结果文件缺失（summary.json 不存在）
│   └── 结果文件解析失败（JSON 格式错误）
│
└── Layer 3：UI 层（Step4AnalyzeViewModel）
    ├── 前置条件不满足
    ├── 用户取消操作
    └── 结果展示异常（数据缺失时的降级显示）
```

## 二、算法层错误处理

### 2.1 错误码处理映射

| 退出码 | 含义 | error_code 对应 | 处理策略 |
|--------|------|----------------|---------|
| 0 | 成功 | 0 | 读取 summary.json，正常流程 |
| 1 | 配置文件错误 | 1 | 记录日志，提示用户重新分析 |
| 2 | 输入文件不存在 | 2 | 提示返回 Step2 重新导入视频 |
| 3 | 视频读取失败 | 3 | 提示视频可能损坏，建议重新导入 |
| 4 | 分析处理失败 | 4 | 记录日志，提示联系技术支持 |
| 5 | 结果导出失败 | 5 | 提示检查磁盘空间 |
| 9 | 未知错误 | 9 | 收集日志，提示联系技术支持 |

### 2.2 三层判断优先级

```
结果判断逻辑：

1. 首先检查 summary.json 是否存在
   ├── 存在 → 读取 success 字段
   │   ├── success = true  → ✅ 分析成功
   │   └── success = false → ❌ 分析失败（从 error_code/error_message 获取详情）
   │
   └── 不存在 → 降级到退出码判断
       ├── 退出码 = 0 → ⚠️ 异常（成功但无结果文件）
       └── 退出码 ≠ 0 → ❌ 根据退出码映射错误信息

2. 补充检查 debug.log（如果存在）
   └── 提取最后 N 行日志作为补充诊断信息
```

## 三、调用层异常处理

### 3.1 进程启动异常

```
场景：exe 文件不存在
│
├── 检查时机：RunAnalysisAsync 开始时
├── 处理方式：
│   ├── 抛出 FileNotFoundException（包含 exe 路径）
│   ├── 上层捕获后显示配置指引
│   └── 引导用户到 设置 → 设备配置 → 算法路径
│
场景：权限不足
│
├── 处理方式：
│   ├── 捕获 UnauthorizedAccessException
│   └── 提示用户检查文件权限或以管理员运行
```

### 3.2 进程超时处理

```
超时处理流程：
│
├── 使用 CancellationTokenSource + CancelAfter(timeout)
│
├── 超时触发后：
│   ├── Step 1: 尝试 process.CloseMainWindow()
│   ├── Step 2: 等待 5 秒
│   ├── Step 3: 如果仍未退出，process.Kill(entireProcessTree: true)
│   ├── Step 4: 检查是否有部分结果（summary.json）
│   └── Step 5: 返回超时错误结果
│
├── UI 提示："分析超时，可能是视频过大或算法异常"
└── 建议操作：
    ├── 缩短分析视频时长
    ├── 增加超时时间配置
    └── 检查系统资源
```

### 3.3 进程异常退出

```
异常退出处理：
│
├── process.ExitCode ≠ 0 且无 summary.json
│   ├── 读取 stderr 缓冲区内容
│   ├── 读取 debug.log（如果存在）
│   ├── 组合错误信息
│   └── 返回 AnalysisResult { Success = false, ErrorMessage = 组合信息 }
│
├── 进程被外部杀死（ExitCode = -1 或平台特定值）
│   └── 提示"分析进程异常终止，可能被外部程序终止"
```

### 3.4 stdout 解析异常

```
stdout 行处理：
│
├── 空行 → 跳过
├── 尝试 JSON 反序列化
│   ├── 成功 → 正常处理
│   └── 失败 → 作为普通日志记录（不抛异常）
│       └── 触发 LogReceived 事件，附带原始文本
```

### 3.5 结果文件异常

```
summary.json 读取：
│
├── 文件不存在
│   └── 根据退出码判断，返回对应错误
│
├── 文件存在但 JSON 格式错误
│   ├── 捕获 JsonException
│   ├── 记录原始文件内容到日志
│   └── 返回"结果文件解析失败"
│
├── 文件存在但关键字段缺失
│   ├── 使用 nullable 类型允许部分缺失
│   └── 缺失的参数在 UI 中显示 "--" 或 "N/A"
│
CSV 文件读取：
│
├── 文件不存在
│   └── 跳过对应图表，UI 中显示"数据不可用"
│
├── 文件格式异常
│   ├── 记录日志
│   └── 跳过对应图表
│
└── 文件过大（>100MB）
    └── 流式读取，避免一次性加载到内存

标注视频文件异常（新增）：
│
├── 文件不存在
│   ├── 跳过视频预览，直接显示参数概览（降级）
│   └── UI 提示"标注视频不可用，仅展示参数数据"
│
├── 文件损坏/无法播放
│   ├── MediaElement.MediaFailed 事件捕获
│   ├── 降级到参数概览视图
│   └── 记录日志，提示用户重新分析
│
├── 视频时长与 CSV 数据时长不匹配
│   ├── 以视频时长为准，CSV 超出部分截断
│   └── 记录日志，Warning 级别
│
└── 视频播放卡顿/性能问题
    ├── 考虑降低 DispatcherTimer 频率（33ms → 50ms）
    └── 减少 Tracker 线更新频率
```

## 四、质量控制策略

### 4.1 质量指标阈值

| 指标 | 优秀 (✅) | 一般 (⚠️) | 差 (❌) |
|------|----------|----------|---------|
| 关键点置信度 | ≥ 0.85 | 0.70 - 0.85 | < 0.70 |
| 有效帧比例 | ≥ 0.95 | 0.80 - 0.95 | < 0.80 |
| 遮挡检测 | 无 | — | 有 |
| 丢点检测 | 无 | — | 有 |

### 4.2 质量等级判定

```
综合质量等级：
│
├── A级（优秀）：所有指标均为 ✅
│   └── UI 显示：绿色标识，"数据质量优秀"
│
├── B级（良好）：有指标为 ⚠️，无 ❌
│   └── UI 显示：黄色标识，"数据质量良好，部分指标需关注"
│
├── C级（较差）：有指标为 ❌
│   └── UI 显示：红色标识，"数据质量较差，建议重新采集"
│
└── 报告中体现：
    ├── 质量等级标识
    ├── 各项指标值 + 状态标记
    └── 如为 C 级，报告中增加醒目提示
```

### 4.3 质量问题处理建议

| 问题 | 可能原因 | 建议操作 |
|------|---------|---------|
| 关键点置信度低 | 光线不足/穿着宽松/背景杂乱 | 调整拍摄环境，重新采集 |
| 有效帧比例低 | 视频抖动/部分帧模糊 | 使用三脚架，重新拍摄 |
| 存在遮挡 | 有物体遮挡受试者 | 清理拍摄区域 |
| 存在丢点 | 关键部位被衣物遮盖 | 调整受试者着装 |

## 五、日志策略

### 5.1 日志分级

```
日志来源：
│
├── 上位机日志（ILogHelper）
│   ├── Information: 分析开始、完成、进度变化
│   ├── Warning: 质量控制警告、可选字段缺失
│   └── Error: 进程异常、文件解析失败
│
├── 算法日志（debug.log）
│   ├── 算法内部处理步骤
│   ├── 异常堆栈信息
│   └── 性能统计
│
└── 任务日志（UI 任务抽屉显示）
    ├── 所有 stdout 状态消息
    ├── 关键上位机日志
    └── 时间戳 + 来源标识
```

### 5.2 日志存储

- 上位机日志：通过现有 `ILogHelper` 框架
- 算法 debug.log：保留在分析结果目录中
- 任务日志：内存中（TaskLogs 集合），不持久化
- 异常诊断包：可导出包含所有相关文件的 zip（后期功能）

## 六、用户取消处理

```
取消流程：
│
├── 用户点击"取消分析"
│
├── 弹出确认对话框："确定要取消当前分析吗？已处理的数据将不会保存。"
│
├── 确认取消
│   ├── 触发 CancellationToken
│   ├── Kill 算法进程
│   ├── 清理临时文件（可选，保留 task_config.json 便于调试）
│   ├── 更新 UI 状态为"已取消"
│   └── 记录取消日志
│
└── 取消后恢复
    └── 显示"重新分析"按钮，允许再次启动
```

## 七、网络/磁盘异常

虽然是本地分析，但仍可能遇到：

| 场景 | 检测方式 | 处理 |
|------|---------|------|
| 磁盘空间不足 | 分析前检查可用空间 | 提示释放空间 |
| 视频文件被锁定 | 尝试打开文件流检测 | 提示关闭占用程序 |
| 输出目录无写入权限 | 尝试创建测试文件 | 提示调整权限或更换目录 |
| 分析过程中文件被删除 | 算法返回 error_code=2 | 引导重新导入 |

## 八、降级展示策略

当部分数据不可用时的 UI 降级方案：

| 缺失数据 | UI 展示 |
|----------|---------|
| 标注视频不存在 | 跳过视频预览，直接进入参数概览视图；隐藏"视频预览"按钮 |
| 标注视频损坏/无法播放 | 视频区域显示"视频无法播放"提示，曲线图仍可独立查看 |
| 步态事件参数某字段为 null | 该参数显示 "--" |
| 运动学汇总全部缺失 | 隐藏运动学区域，显示"运动学参数未计算" |
| CSV 文件不存在 | 视频预览中隐藏对应曲线图；报告中显示"曲线数据不可用" |
| CSV 文件格式异常 | 隐藏对应图表，日志记录详细错误 |
| CSV 与视频时长不匹配 | 以视频时长为准，曲线图 X 轴范围对齐视频时长，超出部分截断 |
| 质量控制信息缺失 | 隐藏质量控制区域 |
| 全部数据缺失 | 仅显示分析状态和基本信息 |

**原则**：能展示多少就展示多少，不因为部分数据缺失而导致整个页面空白。
